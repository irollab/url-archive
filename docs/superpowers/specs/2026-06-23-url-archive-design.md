# URL Archive 设计文档

- **日期**：2026-06-23
- **状态**：已通过头脑风暴评审，待用户复核
- **一句话定位**：一个把浏览器收藏夹从"只写不读的黑洞"变成"会主动帮你思考的副驾驶"的工具——浏览器扩展负责抓取，伴生 Obsidian 插件负责用 AI 思考。

---

## 1. 问题与核心洞察

### 痛点
- **吃灰**：收藏只要一秒，取回却要靠人脑记忆 + 手动翻找；收藏越多取回成本越高，于是不再取回。
- **找不到**：浏览器自带收藏夹要滑很多，标签/文件夹要求用户记得当初怎么归类。
- **插件在大量收藏下失效**。

### 根本矛盾
收藏夹是一个 **"只写不读"的黑洞**：保存成本极低、取回成本极高。

### AI 时代的跃迁
传统方案让**人去适应机器的组织方式**（文件夹、标签）。AI 原生方案反过来——让**机器适应人脑的回忆方式**（模糊语义、意图）。取回方式从 **"导航 / 记忆"** 升级为 **"描述 / 对话"**。

---

## 2. 总体架构

产品由两个组件构成，一条数据流贯穿四层能力：

```
扩展抓（地基）→ 插件建索引（取回）→ 对话编排（RAG）→ 后台触发（复活）
```

| 组件 | 职责 | 分发方式 |
|------|------|----------|
| **浏览器扩展** | 抓取：一键剪藏、正文提取、调 BYOK LLM 生成摘要/标签、写入 vault | Chrome/Edge 应用商店（审核快） |
| **伴生 Obsidian 插件** | 大脑：语义索引、对话取回、RAG 问答、主动复活推送 | 先用 BRAT 分发，并行提交官方商店 |

**存储底座 = Obsidian vault**：一堆本地 markdown。Obsidian 白送了存储、隐私、本地优先、双链/图谱（涌现结构）、可扩展性。

### 关键架构决策
1. **不重复造轮子**：纯"剪藏"已被官方 Web Clipper 与现有 AI 插件覆盖。本产品护城河在**面向"URL 收藏"场景缝合四层能力的智能层**，尤其是别人做得最差的"主动复活防吃灰"。
2. **审核不阻塞**：官方社区插件审核慢（数周到数月），但通过 **BRAT** 可立即向真实用户分发并自动更新；手动安装用于开发调试。官方商店仅为曝光，锦上添花。
3. **BYOK**：所有 LLM 调用使用用户自带 API Key，零运营成本、无服务器、隐私可控。

## 3. 数据模型（地基的地基）

每次剪藏在 vault 中生成**一个 markdown 文件**，默认置于 `URL Archive/` 文件夹。

```markdown
---
url: https://example.com/article
title: 文章原标题
clipped: 2026-06-23T14:30:00      # 收藏时间
domain: example.com
summary: AI 生成的一句话摘要        # 取回时的核心检索字段
tags: [前端, 动画库, 工具]          # AI 自动多维标签
why: 留着做落地页动画选型用          # 用户的"为什么收藏"（可空）
status: unread                     # unread / read / archived
revived: 0                         # 被主动复活提醒过几次（防吃灰）
last_visited:                      # 最近一次从收藏点开的时间
ai_pending: false                  # AI 处理是否待补（离线/失败兜底）
---

> [!summary] 速览
> {{AI 生成的 3-5 句要点摘要}}

## 我的备注
{{why 的展开，或用户手写}}

## 正文快照
{{Readability 提取的正文 markdown —— 防 404 腐烂}}
```

### 设计要点
- **frontmatter 机器读，正文人读 + AI 索引**。`summary` / `tags` / `why` 是语义取回的命中重灾区，放最显眼处。
- **`why`（意图）是产品灵魂**——人回忆时记的是意图。用户不填时 AI 据正文推测。
- **`status` / `revived` / `last_visited` 为第 4 层"防吃灰"预埋**，地基阶段建好字段，上层不再回头改数据结构（避免迁移痛苦）。
- **正文快照内置**，收藏永不失效。
- **一文件一收藏**，契合双链与图谱，标签/`[[]]` 让结构自动生长。

## 4. 第 1 层：地基（剪藏 + AI 处理 + 桥）

### 4.1 剪藏流程

```
点扩展图标 / 快捷键
  → ① 抓当前页：URL、标题、选中文本
  → ② 页面内 Readability 提取正文 → markdown
  → ③ 弹轻量浮层（0.5 秒可划走）：
        ✓ 已剪藏到 Obsidian
        为什么留它？(可选) [______]    ← why，不填也存
        #前端 #动画库 #工具 (AI建议)     ← 可点掉
  → ④ 后台异步调 LLM 生成 summary + 要点 + tags
  → ⑤ 组装 markdown → 经桥写入 vault
```

**要点**
- **先存后理解**：第③步立即给"已保存"确定感；AI 处理异步回填 frontmatter，绝不让用户等 AI。
- **极低摩擦**：`why` 输入框默认聚焦但可秒划走。
- **失败兜底**：AI 挂/离线照样存（`summary`/`tags` 暂空、`ai_pending: true`），有网时补处理。**绝不因 AI 不可用而丢收藏。**

### 4.2 AI 处理：BYOK + 统一适配接口
- 抽象 **`LLMProvider` 接口**，底层走 **OpenAI 兼容协议**（`/chat/completions`、`/embeddings`），一份代码同时支持 OpenAI、Claude（兼容端点）、智谱、DeepSeek、本地 Ollama。未来加"本地模型"近零成本（KISS + 开闭原则）。
- **Key 存储**：扩展 `chrome.storage.local`；绝不进 vault、绝不上传任何服务器。
- **一次调用拿全**：摘要、要点、标签用一个 prompt 返回 JSON：`{ summary, highlights[], tags[] }`，省 token 与往返。

### 4.3 桥（扩展 → vault）
默认走 **Local REST API**（已上架社区插件，无审核风险）：

```
扩展 ──HTTP POST (Bearer Token)──▶ Obsidian Local REST API (localhost:27124)
                                          → 写入 vault 对应 .md
```

**要点**
- **可插拔**：定义 `VaultWriter` 接口，先实现 `RestApiWriter`；未来换自建插件接口或 URI 方案只换实现（依赖倒置）。
- **连不上兜底**：Obsidian 未开/未装插件时，剪藏先进扩展本地队列（IndexedDB），检测可用后自动补写。
- **配置项**：目标文件夹、文件命名规则（默认 `{domain}-{标题slug}-{日期}.md`）、Local REST API 地址与 token。

## 5. 第 2 层：语义取回（解决"找不到"）

住在伴生 Obsidian 插件中。

- **本地向量索引**：遍历 `URL Archive/` 笔记，对 `summary + tags + why + 正文` 生成 embedding，存本地索引（orama 或 sqlite-vss），落在插件私有目录，不污染笔记。
- **检索入口**：Obsidian 命令面板 / 扩展弹层搜索框，自然语言输入 → 向量召回 → 列出命中笔记。
- **混合检索**：向量召回 + frontmatter 关键字过滤（tag / domain / status）。
- **增量索引**：监听文件变化，只重算改动笔记。
- **embedding 走 BYOK**：复用 `LLMProvider` 的 `/embeddings`。

---

## 6. 第 3 层：收藏夹 RAG 问答（私人知识库）

在第 2 层索引上加**对话面板**（Obsidian 右侧 sidebar）：

```
用户提问 → 向量召回 Top-K 相关收藏 → 拼进 prompt
        → LLM 综合作答 + 附来源笔记 [[链接]]
```

- **强制引用来源**：每条回答挂 `[[笔记链接]]`，可点回原文——可信、可追溯的"只读过你认可内容的私人 ChatGPT"。
- 复用第 2 层召回 + 第 1 层 `LLMProvider`，本层几乎只是编排，无新基础设施（YAGNI）。

---

## 7. 第 4 层：主动复活防吃灰（最大差异化）

伴生插件侧的三个后台触发器，全部复用预埋字段与第 2 层索引：

1. **场景化浮现**：用户写/打开某笔记时，用当前内容做向量查询，侧边推"你收藏过这 3 条相关的"。
2. **沉睡提醒**：定期扫 `last_visited` 为空且 `clipped` 超过 N 天的收藏，生成"待复习/待清理"清单；`revived` 计数防重复打扰。
3. **每周回顾**：在每日/每周笔记自动插入"本周值得一看的 5 条沉睡收藏"。

---

## 8. 分阶段实施顺序

每层依赖下层数据，顺序基本固定：

1. **阶段 1 — 地基（MVP）**：扩展剪藏 → 结构化 markdown（含摘要、标签、why、快照）写入 vault。`LLMProvider`（BYOK）+ `VaultWriter`（Local REST API）+ 离线兜底队列。
2. **阶段 2 — 取回**：伴生插件 + 本地向量索引 + 语义/混合检索。
3. **阶段 3 — RAG 问答**：对话面板 + 带来源编排。
4. **阶段 4 — 复活**：三个后台触发器。

---

## 9. 贯穿原则

- **先存后理解、绝不丢收藏**：任何环节（AI、网络、Obsidian 不可用）都不阻塞保存。
- **数据模型一次定对**：预埋上层所需字段，避免迁移。
- **可插拔接口**：`LLMProvider`、`VaultWriter` 隔离外部依赖，便于替换与测试（依赖倒置）。
- **复用而非重造**：上层一律复用下层索引与接口（DRY / YAGNI）。
- **隐私与零成本**：BYOK + 本地存储，无服务器。
